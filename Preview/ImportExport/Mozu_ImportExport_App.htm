Welcome to the Mozu Import/Export Tool

The Mozu Import/Export Tool enables you to import and export product, customer, and orders data to and from Mozu. For new merchants joining the Mozu platform, this tool is a powerful onboarding aid, as it helps you transfer all your existing store data to Mozu, without digging into the guts of the Mozu API. The tool essentially takes all the possible inputs you need to get up and running with Mozu and exposes them through easy-to-read template files.

For merchants already on the platform, the Mozu Import/Export Tool also greatly simplifies batch updating of data by allowing you to export existing data, make the required updates in the template files, and re-import your updated data back into Mozu. You can also use exports as a form of data back-up, and third-party applications can also use exported data for reporting and administrative functions.

Application Features:

  • Exports data related to Products, Customers, and Orders to easy-to-read and modify template (Excel) files.
  • Imports data related to Products and Customers.
  • Dynamically updates exported files to include any new attributes created since the previous download.
  • Enables batch updates of data through a process of Export > Edit > Import.
  • Tracks all import/export operations and creates log files detailing successes and failures.

Install the Mozu Import/Export Tool on Your Tenant

The Import/Export Tool by Mozu is provided for free. Call your sales or professional services representative and request installation on your tenant. Mozu staff will install the app to your tenant, making it available for use in Mozu Admin.

Configure the App

Configuration Requirements

Review the following requirements to ensure you have everything you need to be successful before using the Mozu Import/Export Tool:

  • The Mozu Import/Export Tool must be installed on your tenant.
  • You must have access to Microsoft Excel to open the template files.
  • You must understand the template files.

Enable the App

You do not need to enable the Mozu Import/Export Tool. You access all tool functionality through the Configuration settings dialog for the app.

Use the App

The Mozu Import/Export Tool uses specifically formatted template files to organize your Mozu data.

Note: You can download sample templates from the Import/Export tool, or you can export existing data to get a properly formatted file.

Every time you export data, the templates update to include new data you have added since the previous export. They also display new columns for any new custom attributes you have added. When you edit this file and re-import it, changes to existing products overwrite the old data in Mozu and new data is added. This process provides a seamless cycle for batch processing multiple entries, so you can break up your data import or update into smaller, more manageable jobs.

An example of the cycle of export to edit to import of data using Products.

First-Time Data Import

If you are new to the Mozu platform and importing data for the first time, you must complete some extra steps ensure your data is imported correctly.

Prepare Your Tenant for Initial Import

  1. Decide and outline your catalog architecture.
  2. Ensure you have access to all existing store data. You must have sufficient knowledge of the data structure to map your existing data to the fields in the Mozu template files.
  3. Use Mozu Admin or the Mozu API to prepare your Mozu tenant structure. The exact structure you need will depend on your catalog architecture, locations, etc., but the minimum requirements are as follows:
    • Master Catalog(s): All products in Mozu are associated with Master Catalogs that contain one or more Catalogs. You need 1 or more Master Catalogs.
    • Catalog(s): You need 1 or more Catalogs within each Master Catalog.
    • Locations: To manage inventory, products must be associated with a Location. You need 1 or more locations.
    • Shipping Settings: When you import products, you include information on how they can be shipped to customers. Configure all your Shipping Settings in Mozu, including zones, carriers, methods, and so on.

Data Import Order

To ensure your data appears and links together correctly in Mozu, you must complete your initial data import in the following order:

  1. Contacts
  2. Orders*
  3. Products
Note: Importing orders is a complex process and must be done via the Mozu API. Mozu strongly recommends contacting your professional services representative if you want to import orders from a previous eCommerce solution to Mozu.

Launch the Import/Export Tool

  1. On the Mozu Admin top bar, select Settings > Applications.
  2. Click Mozu Import/Export (version 1.0.0).
  3. Click the Configuration link to open configuration settings.

Export Data

Once you have prepared your tenant correctly, you are ready to export data from Mozu. Even if you have not added real data yet, exporting is the recommended way to get the template files. Performing an export ensures that you are aware of all existing data, and that all your data attributes are reflected in the templates.

Tip: You also can use exports as a form of data backup. Note that importing back-up data that matches existing Mozu data will overwrite the existing data.
  1. Launch the Mozu Import/Export tool.
  2. Select the Export tab.
  3. Select the type of data you want to export from the drop-down menu.
  4. (Optional) For Orders, you can filter the export data by specifying a date range and status.

    Products and Contacts always export all data.
  5. (Optional) Enter an email address to receive a notification when the job completes.
  6. Click Export.
  7. The tool schedules your job. If it is the only job, the export begins immediately.
  8. (Optional) Select the Jobs tab to view all scheduled jobs. You can schedule up to 5 jobs at a time.
  9. When the export is complete, go to the Downloads tab to download the file.
Tip: Refer to the Template Files section of this document for help understanding the data in your export files.

When the job is complete (or if it fails), you can go to the Job Logs tab to view and download information about the job.

Import Data

After adding and editing data in the template files, you can import the data back into Mozu.

Note: Importing data that matches existing data in Mozu overwrites the existing data. This is intended behavior, and necessary for batch updating.
  1. Launch the Mozu Import/Export tool.
  2. Select the Import tab.
  3. Select the type of data you want to import from the drop-down menu.
  4. Click Browse and navigate to the file you want to upload. The file can have any name, but Mozu recommends using some form of identifying information in the name. This is especially helpful if you are scheduling multiple jobs, such as if you are breaking up a large data import into multiple smaller imports.
  5. (Optional) Enter an email address to receive a notification when the job completes.
  6. Click Import.
  7. The tool schedules your job. If it is the only job, the import begins immediately.
  8. (Optional) Select the Jobs tab to view all scheduled jobs. You can schedule up to 5 jobs at a time.

When the job is complete (or if it fails), you can go to the Job Logs tab to view and download information about the job.

Downloading Files

The Downloads tab lists all the files you have imported or exported in the past 3 days. You can click the Download link in the Action column to download a copy of a file.

Note: Files older than 3 days are deleted for storage reasons. Be sure to download any files you want to keep.

The Downloads tab lists both imported and exported files.

Viewing Logs

The Mozu Import/Export Tool creates log files for every import or export you run. When a job completes, you can view these logs on the Job Logs tab.

  1. Launch the Mozu Import/Export tool.
  2. Select the Job Logs tab.
  3. Locate the job you want to view.
  4. Click View Log. The log displays information about all the data objects included in the job, including the date and time they were processed, any updates made, and any errors that occurred. When possible, errors include a CorrelationID that you can use to get more information about the error.

Correlation IDs

In Mozu, a Correlation ID is a unique identifier of an API request. When you import or export data, the Mozu Import/Export Tool makes the API requests necessary to add or update all the data in your files. If your log file for a job includes errors with Correlation IDs, you can use the Correlation ID Lookup in Mozu Dev Center to get more information about the errors:

  1. Log in to Mozu Dev Center and click Console.
  2. Select Logs > Correlation ID Lookup to display the Error Log Information page.
  3. Copy a CorrelationID from your log file and paste it in the field on the right side of the page.
  4. Press the <Enter> key to display any captured data for the ID.

Import/Export Template Files

The Mozu Import/Export Tool splits Mozu data into three broad categories:

  • Products
  • Contacts
  • Orders

You can export all three of these data sets, but you can only import Products and Contacts.

Note: Importing orders is a complex process and must be done via the Mozu API. Mozu strongly recommends contacting your professional services representative if you want to import orders from a previous eCommerce solution to Mozu.

The Import/Export template files are simply Excel files with columns that map to properties and attributes within Mozu. You can map every column in a template file to a member of the Mozu API. In most cases, you can also find a corresponding field in Mozu Admin. This is why it is useful to have a good understanding of your catalog architecture (and how that architecture appears in Mozu Admin) before trying to work with these files.

Data Preparation

Each spreadsheet includes a set of sheets accessible along the bottom of the Excel spreadsheet. Select the sheet and enter data per column. Each row represents a product or contact. Each column is the database data in Mozu.

You can use these templates to batch process new or edit existing products and contacts. You can also use these templates to transfer data from a current database from your current systems into the new Mozu system.

During an import, the system validates only for unique values for specific entries within the sheets. If an ID or ProductCode is reused, and edits exist, content in Mozu will be overwritten from the Excel content. When entering data into the spreadsheets, the following values must be unique:

When entering descriptions and lengthy textual content, be aware of any HTML tags and formatting. The tool and process does not strip any characters, tags, punctuation, or code from your content. If you strip all code and tags prior to adding the content to the Excel sheet, you may sidestep any issues that may occur with displayed formatting later. All content for your products and contacts is styled by theme templates, pages, and CSS through the SiteBuilder in Mozu.

Products Template

The Products template is the largest of the import/export template files, and is likely the one you will use the most. Together, the sheets of the Products template represent your entire catalog structure for all the sites on your tenant. You can use this template to add or update master catalogs, sub-catalogs, product types, product attributes and options, images, inventory, and discounts.

As you work with this file, you will enter information about a single product across multiple sheets. Mozu uses a combination of IDs, codes, names, and catalog information to make the proper data connections. For example, most sheets require a ProductCode to bind the data the sheet contains to the correct product.

Tip: You do not have to include every sheet in the template every time you import data. However, some sheets do have dependencies on others within the template. Required pairings are noted below.

Each of the following sections describes a sheet in the Products template. For each sheet, we define each column, describe valid values for the column, and indicate whether it is required or optional. The sheets in the file appear in the order in which you create product data in Mozu. First you create your attributes, then you create your product types, and so on. For additional context about the process of building out a catalog, log in to Mozu Admin and refer to the Catalog Overview in the Admin Help.

Note: In the following tables, a Required value of No indicates the column is not required for a successful import operation. Some values are not required for an import, but are required Mozu data. For example, you do not have to specify an attribute's InputType every time you modify the attribute, but all attributes must have an InputType to be valid in Mozu.

If you have questions or concerns specific to your data, please contact your Mozu integration and implementation associates.

When using the Product template, you must complete the following sheets according to the data you add:

  • Products: The sheets for attributes, product types, and categories must be populated. Populate the Products, ProductCatalogs, ProductOptions, ProductExtras, ProductBundles, and Images with data.
  • Inventory Amounts: The LocationInventory sheet provides tracking and inventory amounts per product code with locations.

Attributes

Corresponding REST API Resource: commerce/catalog/admin/attributedefinition/attributes

Required With: AttributeValues

Product attributes are the variable properties that define a product. Attributes can be options (such as size or color), properties (such as the brand of the product), or extras (such as custom printing). Because many products share common attributes, you define general attributes before you apply them to specific products.

AttributeValues Sheet

Corresponding REST API Resource: commerce/catalog/admin/attributedefinition/attributes

Required With: Attributes

Attribute values define what customers see for a given attribute. A single attribute can have multiple values. For example, if an attribute has an InputType of List in the Attributes sheet, every item in the list is a value of the attribute. Attribute values are mapped to attrtibues by the AttributeCode.

Column Name Description Valid Values Required*
MasterCatalogName The name of the master catalog that includes the attribute. All attributes in Mozu are defined on the master catalog level, and populate down to sub-catalogs within the master catalog. A string matching the name of the master catalog as it appears at Settings > System Administration > Catalog and Site Structure in Mozu Admin. Yes
AttributeCode A unique code that identifies the attribute. Once an attribute is created, you cannot change its AttributeCode. Any string. The string must be unique to a single attribute in the master catalog. Yes

This value must be unique within the vocabulary for a single attribute and match the DataType of the attribute. If an attribute has multiple values, each Value is specified on a separate row in this sheet. A string or number. The value cannot contain spaces or special characters. en-GB The description for the value in British English. This string defines how the Value appears to customers on the storefront and users in Mozu Admin. Any string. No

ProductTypes Sheet

Corresponding REST API Resource: commerce/catalog/admin/attributedefinition/producttypes

Required With: ProductTypeAttributes, ProductTypeAttributeValues

In Mozu, a product type is a template of settings and attributes you create for a specific set of products. Every product in your catalog has a single product type associated with it. The ProductTypes sheet simply specifies the usage type(s) and master catalog for the product type. You specify details for product types on the ProductTypeAttributes and ProductTypeAttributeValues sheets.

Note: A single product type can support multiple usage types. For example, a shirt can be Configurable and a Component of a larger Bundle.
Column Name Description Valid Values Required*
MasterCatalogName The name of the master catalog that includes the product type. All product types in Mozu are defined on the master catalog level, and populate down to sub-catalogs within the master catalog. A string matching the name of the master catalog as it appears at Settings > System Administration > Catalog and Site Structure in Mozu Admin. Yes
ProductType The name of the product type. This name will appear whenever a user is working with product types in Mozu Admin, so it should be meaningful and human-readable. Any string. Yes
Standard Specifies whether the product type supports the Standard usage type. Standard products are not configurable and do not have variations. A Boolean value, specified as Yes or No. No
Configurable Specifies whether the product type supports the Configurable usage type. Configurable products allow the user to specify a product variation based on product options. For example, a shirt in size Small. A Boolean value, specified as Yes or No. No
Bundle Specifies whether the product type supports the Bundle usage type. Bundles are collections of products sold as a single entity. For example, a suit Bundle can contain a shirt, pants, and a jacket. A Boolean value, specified as Yes or No. No
Component Specifies whether the product type supports the Component usage type. Components can belong to Bundles. For example, a shirt can be a Component of a suit Bundle. A Boolean value, specified as Yes or No. No
GoodsType The type of goods the product type contains. This value distinguishes between physical items, digital items, and digital store credits (such as gift cards). The default is Physical. A string matching one of the following:
  • Physical
  • Digital
  • DigitalCredit

ProductTypeAttributeValues Sheet

Corresponding REST API Resource: commerce/catalog/admin/attributedefinition/producttypes

Required With: ProductTypes, ProductTypeAttributes

The attribute values you specify for a given product type define what users and customers see for that product type. These values might differ from the values defined on the AttributeValues sheet. For example, your store might include a line of shirts that are only available in sizes Small, Medium, and Large. However, your Size attribute from the AttributeValues sheet has possible values of XS, S, M, L, and XL. You can still use your Size attribute for the shirts by giving them their own ProductType and only enabling the three relevant values for Size on this sheet.

Categories Sheet

Corresponding REST API Resource: commerce/catalog/admin/categories

Required With: NA

In Mozu, you uses product categories to group similar items so that you can display them together in sections on your site. Categories help you create faceted navigation for your customers. For example, Women's Apparel can be its own category as well as a parent cateogry of Shirts, Pants, and Dresses. Categories map to catalogs, and you can define them on the master catalog or only on a specific sub-catalog. A single product can belong to multiple categories.

IsDisplayed Specifies whether the category is visible to customers on the storefront. A Boolean value, specified as Yes or No. No
LocaleCode The locale code used to decide the language in which to display category content, in standard ISO language-country format. An ISO locale code. For example, en-US.

Currently, en-US is the only supported value.		 No
MetaTagTitle The metadata title for the category page, in the language specified by LocaleCode. This title is not visible to customers, but search engines may index this value, so using a good title can improve your SEO. Any string. No
MetaTagDescription The metadata description for the category page, in the language specified by LocaleCode. Search engines may display this description in search results. Any string. No
PageTitle The title that customers see when they land on the page, in the language specified by LocaleCode. This can be different than the MetaTagTitle, but should still be a logical term that customers might search. Any string. No
CategoryDescription The description of the category as it appears in Mozu Admin. Depending on your theme, you can also make this description visible to customers on your storefront. Any string. No

SEOUrl The SEO friendly URL for the category page. This is the slug that customers see when they navigate to the page. For example, if you want the page URL to appear as "mystore.com/shirts", you can enter shirts for this value. Any string. No

Products Sheet

Corresponding REST API Resource: commerce/catalog/admin/products

Required With: ProductCatalogs, ProductOptions, ProductExtras, ProductBundles, ProductImages

Once you have creates attributes, product types, and categories, you can add products to your store. Do not try to add products to your store before building out the catalog infrastructure that the previous tabs describe. Your products will be missing required data and the import will not work. However, once your catalog infrastructure is set, you can fill out just the product-related sheets when you are adding or updating product data.

Note: In addition to the columns described in the following table, the Products sheet includes columns for all the attributes you create. To apply an attribute to a product, enter a valid value for the attribute in that attribute's column. Leave the attribute empty if you do not want to apply it to the product. If you are creating new attributes in the same import that you are adding them to products, you can add the attribute code to the end of this sheet as a column header.

ProductTypeName The name of the product type associated with the product. Any string that is a valid product type name. This value should match a ProductType on the ProductTypes sheet. Yes

  • Standard

  • Bundle
  • Component

Contacts Template

The following information gives a description of each column in the sheets of the Contacts import/export template. You can use this information to map data for your customer accounts and contacts. Customer accounts include customer shoppers and Mozu Admin users.

Contacts include customer accounts and contact information. This data is used by Mozu for all shopper accounts and contacts to manage orders and contact information for subscribed messages, promotional emails, newsletters, and more.

Associations between data are defined by entered IDs, codes, and names through the sheets. For example, you may have multiple entries associated to the same AccountId for defining all available values.

All Booleans should have Yes or No value entered. If you have questions or concerns with the exact required values and options specific to your data, consult your Mozu integration and implementation associates assigned to your account.
